## Core Concepts

### 1. Pod Lifecycle

A Pod goes through multiple states:

```
Pending → Scheduled → Running → Succeeded / Failed
```

If a pod is stuck in `Pending`, check node resources or scheduling constraints (taints, tolerations, resource limits).

---

### 2. Liveness vs Readiness Probes

| Probe | Purpose |
|---|---|
| **Readiness** | Controls whether pod receives traffic |
| **Liveness** | Controls whether pod should be restarted |

**Strong answer line:**
> "Readiness controls traffic routing; liveness controls container restart behavior."

---

### 3. Service Types

| Type | Use Case |
|---|---|
| ClusterIP | Internal pod-to-pod communication |
| NodePort | Expose via node IP (dev/testing) |
| LoadBalancer | External access via cloud provider LB |

---

### 4. Ingress Flow

```
User → Load Balancer → Ingress Controller → Service → Pod
```

Ingress routes external HTTP/HTTPS traffic to services using host/path rules.

## Standard Debugging Flow

```bash
# 1. Check pod status
kubectl get pods

# 2. Get detailed events and state
kubectl describe pod <name>

# 3. Check application logs
kubectl logs <pod>

# 4. Exec into container for live debugging
kubectl exec -it <pod> -- sh
```

**Framework:**
1. Check pod status
2. Describe the pod to look for events
3. Check logs
4. Exec into the pod to debug inside the container

## Interview Questions + Answers

**Q1: Pod is not starting. What do you do?**

```bash
kubectl get pods
kubectl describe pod <name>
```

Look for:
- Image pull errors (`ImagePullBackOff`)
- Resource quota issues
- Crash loops (`CrashLoopBackOff`)
- Scheduling failures (node affinity, taints)

---

**Q2: Pod is running but app not accessible.**

Check:
- Readiness probe — is it passing?
- Service selector — does it match pod labels?
- Port configuration — does `targetPort` match `containerPort`?
- Ingress rules — correct host/path?

```bash
kubectl get endpoints <service>
# If empty → label mismatch between service selector and pod labels
```

---

**Q3: Difference between ClusterIP and LoadBalancer?**

- `ClusterIP`: internal-only, accessible within the cluster
- `LoadBalancer`: provisions a cloud provider LB, accessible externally

---

**Q4: How does scaling work?**

```bash
# Manual
kubectl scale deployment my-app --replicas=5

# Automatic (HPA)
kubectl autoscale deployment my-app --cpu-percent=70 --min=2 --max=10
```

Horizontal Pod Autoscaler scales based on CPU, memory, or custom metrics.

---

**Q5: What happens when a pod crashes?**

Kubernetes restarts the container based on `restartPolicy`. If it keeps failing, it enters `CrashLoopBackOff` — backoff delay increases with each restart attempt.

---

**Q6: How does Kubernetes networking work?**

Every pod gets its own IP address. Pods can communicate directly within the cluster. Services provide stable endpoints and load balancing across pod replicas. CoreDNS provides service discovery.

---

**Q7: What is Ingress?**

Ingress is a Kubernetes API object that routes external HTTP/HTTPS traffic to internal services based on rules (host or path). It requires an Ingress Controller (e.g., nginx-ingress, GKE's built-in) to actually handle traffic.

---

**Q8: Common issues in Kubernetes?**

- `CrashLoopBackOff` — app crash, bad config, missing env vars
- `ImagePullBackOff` — bad image tag, registry auth issue
- Service misconfiguration — label mismatch, wrong port
- DNS issues — CoreDNS failure, wrong service name
- Resource limits — pod evicted, OOMKilled

## Real Scenario Debugging

### Scenario 1: CrashLoopBackOff
```bash
kubectl logs <pod>               # Check app errors
kubectl describe pod <pod>       # Check events
# Fix: check env vars, app config, missing dependencies
```

### Scenario 2: Service Not Reachable
```bash
kubectl get endpoints <service>  # Should not be empty
kubectl get pods --show-labels   # Verify labels match selector
kubectl describe svc <service>   # Check port config
```

### Scenario 3: Ingress Not Working
```bash
kubectl describe ingress <name>  # Check rules and backend
kubectl get pods -n ingress-nginx  # Check controller status
# Also verify: DNS resolution, TLS cert if HTTPS
```

## Hands-On Practice: GKE Lab

### Step 1: Create Cluster
```bash
gcloud container clusters create dev-cluster \
  --zone us-central1-a \
  --num-nodes 2
```

### Step 2: Connect
```bash
gcloud container clusters get-credentials dev-cluster --zone us-central1-a
```

### Step 3: Deploy App
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: app
        image: nginx
        ports:
        - containerPort: 80
```

```bash
kubectl apply -f deployment.yaml
```

### Step 4: Expose via Service
```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  type: LoadBalancer
  selector:
    app: my-app
  ports:
  - port: 80
    targetPort: 80
```

```bash
kubectl apply -f service.yaml
kubectl get svc  # Wait for external IP
```

### Step 5: Break + Fix Exercises

**Break 1: Wrong targetPort**
```yaml
targetPort: 3000  # Container runs on 80
```
Debug: `kubectl get endpoints` → endpoints exist but connection fails

**Break 2: Wrong labels**
```yaml
selector:
  app: wrong-label  # Doesn't match pod labels
```
Debug: `kubectl get endpoints` → empty

**Break 3: Bad image**
```yaml
image: nginx:invalid-tag
```
Debug: `kubectl describe pod` → `ImagePullBackOff` with registry error
